ABSTRACT

GigSpace is a web-based freelance marketplace platform that connects clients who need services with freelancers who provide those services. The system supports the two core user roles—clients and freelancers—and provides end-to-end workflows such as project posting, bidding, hiring, order management, and communication through an integrated messaging system. The platform also supports freelancer gigs (direct service offers), profile management, skill and category browsing, file uploads, and dashboard-based user experiences.

The application is implemented using PHP and MySQL (XAMPP environment) with session-based authentication. Database persistence is achieved with a relational schema defined in gigspace.sql. The system uses server-side page rendering with modular components and AJAX-style process endpoints for actions like placing bids, accepting/rejecting proposals, sending messages, retrieving conversations, and managing order status.

This report documents the project motivation, requirements, system design, database structure, implementation approach, testing plan, security considerations, challenges encountered, and proposed future improvements.

1. INTRODUCTION

Freelancing has become one of the fastest-growing alternatives to traditional employment, enabling individuals and organizations to source specialist services remotely. Freelance platforms reduce effort by centralizing service discovery, facilitating agreement workflows, and providing communication tools.

GigSpace is designed as a practical student/academic-scale freelance marketplace application, with a focus on providing a realistic workflow rather than a simple CRUD application. The system aims to support a complete transaction lifecycle: clients post projects, freelancers bid or offer gigs, an order is created, both parties communicate and share delivery items, and the process can end with completion and review.

The application runs on a local XAMPP environment using PHP and MySQL. It supports media-rich user profiles (images), messaging threads linked to orders, and skill-based discovery.

2. PROBLEM STATEMENT

Many freelance marketplace systems are either too complex, expensive to implement, or missing essential workflow features such as:

- Structured bidding and hiring flows

8. SYSTEM DESIGN

8.1 High-Level Architecture

GigSpace follows a web application structure with:

- Presentation layer: PHP page templates (root pages and dashboards)
- Logic layer: process endpoints under /process that handle AJAX-style requests or POST actions
- Data layer: MySQL database accessed via mysqli in config/db_config.php

File-based routing is used: individual PHP files are served as endpoints (e.g., index.php, hire.php, projects.php, order.php, freelancers.php).

8.2 Functional Modules

The system can be separated into modules:

A) Authentication & User Management
- Registration for clients and freelancers
- Login/logout
- Session-based user_id and role tracking

B) Profiles and Skills
- Public profiles via view_freelancer.php
- Profile images stored in dashboards/uploads/profile/
- Skill association using freelancer_skills join table

C) Project & Gig Discovery
- Categories and skills browsing
- Featured freelancers section on the homepage

D) Bidding Workflow (Client Projects)
- Place bid: process/place_bid.php
- Accept bid: process/accept_bid.php
- Reject bid: process/reject_bid.php

E) Order Lifecycle
- Order created after bid acceptance or gig hiring
- Order status updates (including delivery stages)
- Delivery note/file support

F) Messaging System
- Conversations per order
- Message sending: process/send_message.php
- Message retrieval: process/get_messages.php
- Unread count retrieval: process/get_unread_count.php

G) Delivery/Submission & Status Update
- Delivery submission: process/submit_work.php
- Update order/project status: process/update_project_status.php

H) Subscription
- Newsletter subscription: process/subscribe.php

8.3 User Roles and Navigation Model

- Clients access client dashboards: projects, gigs browsing, messages, orders, profile
- Freelancers access freelancer dashboards: gigs, projects, messages, earnings, profile

Navigation and page access depend on role checks using $_SESSION.

10. DATABASE DESIGN

10.1 Database Overview

GigSpace uses a relational MySQL database defined in gigspace.sql. The database is normalized into tables representing:

- Authentication users
- Public profiles
- Freelancer-specific profiles
- Skills and category taxonomy
- Projects, bids, gigs, orders
- Conversations and messages
- Reviews and attachments

10.2 Entity Descriptions (Tables)

The primary entities are:

1) users
- Stores username, email, password hash, role (client/freelancer)
- Tracks account state (is_active, is_verified)

2) profiles
- Stores full_name, phone_number, profile_image, bio, country, university
- Linked to users.user_id

3) freelancer_profiles
- Extends freelancer users with title, experience_level, rating, total_reviews, completed_orders

4) skills
- Skill catalog

5) freelancer_skills
- Join table mapping freelancer_profiles to skills

6) categories
- Job categories (e.g., Web Development, Graphic Design)

7) projects
- Client project postings including budget_min/budget_max, deadline, status

8) gigs
- Freelancer direct services with price, delivery_time, revisions, status

9) bids
- Proposals placed by freelancers on projects with amount, proposal, delivery_time, status

10) orders
- Transaction representation that includes either gig_id or project_id
- Holds requirements, delivery notes/file urls, total_price, status, timestamps, paid_at

11) conversations
- Represents chat threads linked to orders

12) messages
- Represents individual messages inside conversations
- Includes message_type and optional attachment_url and is_read flag

13) reviews
- Reviews per order with rating and comment

14) gig_images
- Images attached to gigs

15) attachments
- Generic attachments linked to users, using related_type/related_id fields

16) subscribers
- Stores newsletter emails

10.3 Relationships and Constraints

The schema includes foreign keys such as:

- profiles.user_id → users.id
- freelancer_profiles.user_id → users.id
- freelancer_skills.freelancer_id → freelancer_profiles.id
- freelancer_skills.skill_id → skills.id
- projects.client_id → users.id
- bids.project_id → projects.id (cascade delete)
- bids.freelancer_id → freelancer_profiles.id (cascade delete)
- orders.gig_id/projects_id/client_id/freelancer_id reference their respective entities
- conversations.order_id → orders.id (cascade delete)
- messages.conversation_id → conversations.id

These constraints ensure referential integrity for core entities, supporting predictable workflow states.

11. IMPLEMENTATION

11.1 Technology Stack

- Backend: PHP (procedural), sessions for authentication
- Database: MySQL (MariaDB compatible), mysqli extension
- Frontend: HTML5/CSS3 (PHP-included styling), responsive UI
- JavaScript: used for AJAX interactions and dynamic updates (where applicable)
- Server: Apache via XAMPP on Windows 11

Project configuration:
- config/init.php: session_start and initialization
- config/db_config.php: MySQL connection credentials

11.2 Key Application Flow

A) Registration and Login
1. User submits signup form through components/signup*.php
2. Server hashes password and stores in users table
3. If freelancer, create freelancer profile record
4. Redirect user to authenticated pages

B) Freelancer Profile & Skills
1. Freelancer uploads profile image into dashboards/uploads/profile/
2. Skills selected are stored via freelancer_skills join table
3. Public profile becomes available for clients

C) Client Project Posting
1. Client creates a project using hire.php and process/create_project.php
2. The project is stored in projects table

D) Bidding
1. Freelancer browses projects and places a bid using process/place_bid.php
2. Bid saved into bids table with proposal and delivery time

E) Accepting/Rejecting Bids
1. Client accepts bid: process/accept_bid.php
2. On acceptance, an order is created (orders table)
3. If rejected, bid status changes to rejected using process/reject_bid.php

F) Order and Messaging
1. Messages are stored in messages table linked through conversations and order_id
2. Client and freelancer dashboards display conversation history
3. Unread counts are retrieved using process/get_unread_count.php

G) Delivery and Completion
1. Freelancer submits work via process/submit_work.php
2. Order status updates (e.g., delivered/completed)
3. Optional review creation in reviews table

11.3 Important Endpoints and Their Responsibilities

- /process/create_project.php
  Handles project creation form data and stores in projects.

- /process/place_bid.php
  Stores a new bid record linked to project_id and freelancer_id.

- /process/accept_bid.php
  Changes bid status to accepted and creates corresponding order and conversation.

- /process/reject_bid.php
  Changes bid status to rejected.

- /process/send_message.php
  Inserts messages associated with a conversation.

- /process/get_messages.php
  Returns conversation messages for UI rendering.

- /process/get_unread_count.php
  Returns the number of unread messages for a user’s conversations.

- /process/submit_work.php
  Stores delivery content such as delivery_file_url or delivery_note.

- /process/update_project_status.php
  Updates order/project workflow states.

- /process/subscribe.php
  Adds email to subscribers table.

12. SECURITY CONSIDERATIONS

This section identifies key areas relevant to a marketplace platform.

Authentication & Authorization
- Session-based authentication controls access to dashboards and actions.
- Role checking prevents clients from accessing freelancer-only workflows.

SQL Injection
- The prototype uses mysqli_* operations; prepared statements would be required for production.
- Input validation should be applied before database queries.

Cross-Site Scripting (XSS)
- User-generated content such as proposals and messages must be escaped on output.

File Upload Security
- Uploaded profile/gig images are stored under dashboards/uploads.
- A production version should implement:
  - MIME/type validation
  - Virus scanning (where possible)
  - Safe extension allow-list
  - Upload size limits
  - Randomized file names to prevent overwriting

CSRF and Rate Limiting
- Production should include CSRF tokens for POST/PUT actions.
- Rate limiting should be applied to login, message sending, and bid placement.

Sensitive Data Handling
- Database credentials in config/db_config.php must be secured and rotated if exposed.
